## SECTION 1 — PROJECT OVERVIEW & VISION

### 1.1 What Is EduQuest?

EduQuest is a full-stack, gamified, minimalist educational web platform designed specifically for Indian students. It serves two primary audiences:

Audience Group 1 — School Students:
- Class 9 students following the CBSE curriculum
- Class 10 students preparing for board examinations
- Class 11 students in Science, Commerce, or Arts streams
- Class 12 students preparing for board and entrance exams

Audience Group 2 — Engineering & Programming Learners:
- Students and beginners who want to learn popular programming languages
- Day-by-day structured plans for each language
- No Go language is included anywhere in this platform

### 1.2 What Makes EduQuest Different?

EduQuest is NOT just another tutorial website. It combines five powerful systems into one:

System 1 — Structured Day-by-Day Learning:
Every subject (for school students) and every programming language (for engineering students) has a complete day-by-day plan. The student knows exactly what to study on Day 1, Day 2, Day 3 and so on until the course is complete.

System 2 — Gamification Engine:
Every action the student takes on the platform earns points. Points fill up a level bar. When the level bar fills, the student levels up. Levels go from 1 to 100. This is inspired by BGMI (Battleground Mobile India) where leveling feels rewarding and progressive.

System 3 — Streak Tracker:
Students are rewarded for coming back to the platform every single day and solving at least one problem. This is inspired by GitHub's contribution graph and LeetCode's daily streak counter. Missing a day breaks the streak.

System 4 — Real-Time Competitive Battles:
Students can enter a matchmaking queue and be matched with another student of the same class and similar level (within ±2 levels). They both pick the same subject and chapter. Then they compete in real-time — fastest and most correct answers wins. This is inspired entirely by the BGMI matchmaking flow.

System 5 — Events & Competition Portal:
Colleges and institutions can apply to host their own competitions directly on the EduQuest platform. The competition runs inside a safe browser environment — no copy-paste, no new tabs allowed.

### 1.3 Core Values

- Fairness: Every student starts at Level 1, regardless of background
- Honesty: No copy-paste, no shortcuts, anti-cheat is built into every question
- Clarity: Minimalist design — no clutter, no confusion
- Progress: Every day feels like growth — points, streaks, levels, community
- Accessibility: Works on mobile, tablet, and desktop with the same quality
- India-First: All pricing is in Indian Rupees (₹), all content is CBSE-aligned

### 1.4 Platform Language

All user-facing text is in English. The platform is designed for Indian students but uses English as the primary language for all instructions, questions, explanations, and UI labels.

## SECTION 2 — DESIGN PHILOSOPHY (MINIMALIST UI RULES)

### 2.1 Core Design Rules

Rule 1 — Minimalism Above All:
The interface must be clean and breathable. Every element on screen has a reason to be there. Nothing is added just to fill space.

Rule 2 — No Confusion:
Every page must be self-explanatory. A new student landing on the platform for the first time must understand what to do within 5 seconds of looking at the screen.

Rule 3 — Eye-Catching Without Being Loud:
The platform should look modern, premium, and attractive. It should use subtle gradients, clean typography, smooth transitions, and meaningful icons — not flashy or distracting elements.

Rule 4 — Consistent Color Language:
- Primary color: Deep Indigo (trust, education, focus)
- Accent color: Vibrant Amber (reward, achievement, warmth)
- Background: Pure White or very light gray (clean, readable)
- Error states: Soft Red
- Success states: Emerald Green
- Dark mode: Deep Navy with Off-White text

Rule 5 — Typography Rules:
- Heading font: Inter or Sora (modern, geometric, clean)
- Body font: Inter (consistent with headings, excellent readability)
- Monospace font for code blocks: JetBrains Mono
- No more than 3 font sizes per page to avoid visual chaos

Rule 6 — Spacing & Breathing Room:
- All cards and sections must have generous padding
- Nothing should feel cramped
- Use whitespace intentionally — it is not empty space, it is a design element

Rule 7 — Mobile First:
Design every screen for a 375px wide mobile screen first, then scale up to tablet (768px) and desktop (1280px).

Rule 8 — One Primary Action Per Screen:
Every page or modal must have exactly one primary call-to-action button. Secondary actions are visually de-emphasized.

### 2.2 Navigation Design

The top navigation bar contains:
- EduQuest logo (left side)
- Navigation links: Home | Class 9 | Class 10 | Class 11 | Class 12 | Engineering | Community | Events (center on desktop, hamburger on mobile)
- Right side: Dashboard icon, notification bell, user avatar (if logged in) OR Sign In / Sign Up buttons (if not logged in)

The navigation is sticky — it stays at the top as the user scrolls.

On mobile: All links collapse into a smooth slide-out side drawer menu.

### 2.3 Page Load Experience

- Every page uses skeleton loading screens (not spinners) — this gives the impression of instant loading
- All transitions between pages use a smooth fade + slight slide animation
- First meaningful content must appear within 1.5 seconds on a 4G connection

## SECTION 3 — COMPLETE MODERN TECHNOLOGY STACK (NO GO LANGUAGE)

### 3.1 Frontend Technology

Primary Framework: Next.js 14 (React-based, App Router)
- Why: Server-side rendering for SEO, fast page loads, file-based routing
- Version: Next.js 14 with App Router (not Pages Router)
- Rendering strategy: Hybrid — SSR for public pages, CSR for dashboard and battles

UI Component Library: Shadcn/UI + Radix UI Primitives
- Why: Accessible, unstyled components that can be fully customized
- All components are headless — no opinionated styling forced on us

Styling: Tailwind CSS v3
- Why: Utility-first, fast to write, consistent spacing/color system
- Custom theme defined in tailwind.config.ts to match EduQuest brand colors

Animation Library: Framer Motion
- Why: Industry-standard for React animations — smooth, declarative, GPU-accelerated
- Used for: page transitions, card reveals, level-up animations, streak fire animations

Icons: Lucide React + React Icons
- Why: Clean, consistent, lightweight SVG icons

State Management: Zustand
- Why: Lightweight global state — simpler than Redux, perfect for auth state, stars wallet, level display

Data Fetching: TanStack Query (React Query v5)
- Why: Server state management, caching, background refresh, optimistic updates

Real-Time Communication: Socket.IO (Client side)
- Why: Live matchmaking battles require persistent bidirectional connection

Charts & Progress Visualization: Recharts + Custom SVG progress rings

Code Editor (for engineering questions): Monaco Editor
- Why: Same editor as VS Code — students feel at home; supports syntax highlighting for all languages

Math Rendering (for Class 11/12 Physics, Chemistry, Math): KaTeX
- Why: Fast LaTeX rendering in browser for equations and formulas

Sound Effects: Howler.js
- Why: Best-in-class audio library for browsers — supports multiple formats, volume control, spatial audio

Safe Browser Implementation: Custom sandboxed iframe + browser extension (for competition mode)
- The safe browser disables: right-click, copy, paste, new tab, developer tools, print screen detection

### 3.2 Backend Technology

Primary Backend Framework: NestJS (Node.js)
- Why: TypeScript-first, modular, scalable, built-in dependency injection
- Structure: Modules for Auth, Users, Classes, Engineering, Battles, Events, Community, Stars, Admin

Runtime: Node.js v20 LTS
- Why: Stable, fast, excellent ecosystem, no Go language needed

API Design: REST API + WebSocket (Socket.IO server)
- REST for standard operations: login, fetch courses, fetch questions, submit answers
- WebSocket for real-time: matchmaking queue, live battles, live leaderboards

Database: PostgreSQL 16
- Why: Robust, ACID-compliant, excellent for relational data like users, courses, battles
- ORM: Prisma (TypeScript-native, auto-generates types from schema)

Caching Layer: Redis 7
- Why: Session storage, matchmaking queue state, leaderboard rankings, rate limiting
- Client: ioredis

Search Engine: Elasticsearch (or Meilisearch for lighter use)
- Why: Fast full-text search across questions, topics, community posts

File Storage: AWS S3 (or Cloudflare R2 for cost savings)
- Used for: user avatars, competition attachments, simulation assets

Email Service: Nodemailer + Amazon SES
- Used for: verification emails, password reset, event notifications

Payment Gateway: Razorpay
- Why: Best payment gateway for Indian market, supports UPI, cards, net banking, wallets
- Used for: ₹200/month subscription, ₹100 skill pack purchases

Authentication: JWT (JSON Web Tokens) + Refresh Token rotation
- Access token: 15 minutes expiry
- Refresh token: 7 days expiry, stored in httpOnly cookie

### 3.3 Infrastructure & DevOps

Hosting (Frontend): Vercel
- Why: Built for Next.js, global CDN, automatic SSL, zero-config deployment

Hosting (Backend): Railway or Render (or AWS EC2 for production scale)

Database Hosting: Supabase (PostgreSQL) or PlanetScale (MySQL alternative)

CDN: Cloudflare
- Why: Free tier is generous, global edge network, DDoS protection

Containerization: Docker
- Why: Consistent environments across development, staging, production

Container Orchestration: Docker Compose (development) → Kubernetes (production scale)

CI/CD Pipeline: GitHub Actions
- On every push to main: run tests → build → deploy

Monitoring: Sentry (error tracking) + Grafana + Prometheus (metrics)

Logging: Winston (structured logging) + Loki (log aggregation)

Environment Management: dotenv + Vault (for production secrets)

## SECTION 5 — COMPLETE FRONTEND FOLDER & FILE STRUCTURE (WITH COMMENT RULES)

### 5.1 Comment Rules (Applied to Every Single File)

Every file in the project — whether it is a page, component, utility, hook, or style file — MUST have a comment block at the very top of the file. This comment block must include:

Comment Format Required for Every File:
```
FILE: [filename]
LOCATION: [full relative path from project root]
PURPOSE: [1-2 sentences explaining what this file does]
USED BY: [which pages or components use this file]
DEPENDENCIES: [what other files/packages this file imports]
LAST UPDATED: [date]
AUTHOR NOTE: [any important notes for future developers]
```

Comment Rules Inside Each File:
- Every function must have a comment above it explaining what it does, what parameters it takes, and what it returns
- Every complex variable or constant must have an inline comment
- Every API call must have a comment explaining what endpoint it calls and why
- Every conditional block (if/else) that is non-obvious must have a comment
- Every loop must have a comment explaining what it is iterating over and why

## SECTION 14 — QUESTION SYSTEM: TYPES, DIFFICULTY, POINT VALUES

### 14.1 Question Types

Type 1 — Multiple Choice Questions (MCQ)
- 4 options (A, B, C, D)
- Only one correct answer
- User clicks their choice then presses Submit
- Cannot unselect after Submit
- Used for: Concept testing, factual recall, formula identification

Type 2 — Multiple Select Questions (MSQ)
- 4 or more options
- More than one correct answer possible
- User must select ALL correct answers to earn full points
- Partial credit: For every correct option selected without selecting a wrong one, half points
- Used for: Testing understanding of multiple related concepts

Type 3 — Short Answer Questions
- User types a word, a number, or a very short phrase (1–5 words)
- Answer is compared against accepted answers list (case insensitive)
- Used for: Fill in the blank, definition recall, formula values

Type 4 — Long Answer Questions
- User types a paragraph or multi-sentence answer
- Marked by an AI model (or admin-reviewed in events)
- Full points for complete correct answers
- Half points for partial answers
- Used for: Explain, describe, analyze type questions

Type 5 — True / False Questions
- Binary choice
- User clicks True or False
- Used for: Conceptual understanding checks

Type 6 — Fill in the Blank Questions
- Text with one or more blanks
- User types into each blank
- Used for: Recall of specific terms, values, names

Type 7 — Code Trace Questions (Engineering only)
- A code snippet is shown
- User is asked: "What will this code output?" or "What is the value of X after line 5?"
- Answer is a short text or number
- Used for: Testing code reading ability without writing code

Type 8 — Code Writing Questions (Engineering only)
- Monaco Editor appears with a starter template
- User writes code to solve the given problem
- Code is submitted and run against test cases
- Output is compared against expected output
- Test case results shown: "Passed 3/5 test cases"
- Points awarded proportionally based on test cases passed
- Copy-paste is disabled inside Monaco Editor

Type 9 — Diagram-Based Questions
- An image or diagram is shown
- Questions are asked based on the diagram (identify parts, label components, predict output)
- Used for: Biology diagrams, Physics circuit diagrams, Geography maps

Type 10 — Match the Columns
- Two columns shown
- User drags items from Column B to match with Column A
- All correct = full points; partial matches = partial points

### 14.2 Question Difficulty Levels

Level 1 — Easy:

## SECTION 19 — BGMI-STYLE REAL-TIME MATCHMAKING & LIVE BATTLE SYSTEM

### 19.1 What the Battle System Is

The Battle System is a 1v1 real-time competitive quiz feature inspired entirely by the matchmaking flow of Battleground Mobile India (BGMI). Two students of the same class and similar level compete against each other in solving questions faster and more accurately.

### 19.2 Who Can Battle Whom (Matching Rules)

Rule 1 — Same Class Only:
- Class 9 students can only battle other Class 9 students
- Class 10 students can only battle other Class 10 students
- Class 11 students only battle Class 11 students
- Class 12 students only battle Class 12 students
- Engineering students battle other engineering students (across any language — but questions are language-specific)

Rule 2 — Level-Based Matching:
- Players are matched within a ±2 level range
- Example: A Level 10 player will only be matched with Level 8, 9, 10, 11, or 12 players
- If no match is found within ±2 in 30 seconds, range expands to ±4 levels
- If still no match in 60 seconds, range expands to ±6 levels
- Maximum wait: 3 minutes — if no match, battle is cancelled and user is notified

Rule 3 — Same Subject & Chapter:
Before entering the matchmaking queue, the student MUST select:
- Which subject (e.g., Mathematics)
- Which chapter (e.g., Chapter 6 — Triangles)
- The system matches players who chose the SAME subject AND chapter

Rule 4 — Level 6 Minimum:
- Battle mode is unlocked at Level 6
- Students below Level 6 cannot enter the battle queue
- This ensures only students with foundational knowledge compete

### 19.3 Pre-Battle Flow (Step by Step)

Step 1 — Click "Battle" in the navigation
Step 2 — Choose your class (auto-filled from user profile — cannot change)
Step 3 — Choose subject from your class
Step 4 — Choose chapter from that subject
Step 5 — Stars Wager Screen (only for Level 10+ users):
  - User can choose to wager 0 Stars (friendly battle) or 10–50 Stars (ranked battle)
  - Maximum Stars wager per single battle: equivalent of ₹50 in Stars
  - Stars daily limit: equivalent of ₹100 in Stars per day (across all battles combined)
  - If user has insufficient Stars, they can only do a friendly battle (0 Stars)
Step 6 — Click "Find Opponent"
Step 7 — Matchmaking Screen:
  - Animated searching screen: pulsing circles, "Finding your opponent..." text
  - Shows: Subject chosen, Chapter chosen, Stars wagered
  - Cancel button available at any time
Step 8 — Match Found:
  - Opponent's username and level are shown briefly (3 second countdown)
  - "Get Ready!" screen with animated countdown 3...2...1...Battle!

### 19.4 Live Battle Room

The battle room screen is divided into two halves:

Left Side (Your Side):
- Your avatar and username at the top
- Your current score in this battle
- A question is shown in the center
- Your answer options below
- Your total correct/wrong count

Right Side (Opponent's Side):
- Opponent's avatar and username
- Opponent's current score (updates in real time via Socket.IO)
- A progress indicator (how many questions they've answered)
- You cannot see their specific answers — only their score

Center:
- Battle timer (total battle is 5 minutes)
- Current question number (e.g., "Question 4 of 10")

### 19.5 Battle Questions

- 10 questions per battle (all from the selected chapter)
- Mix: 5 Easy, 3 Medium, 2 Hard
- Questions are different for each battle (randomly selected from the question bank)
- No hints — battles are pure knowledge testing
- No YouTube links — this is a competition
- Copy-paste is blocked (same anti-cheat as everywhere else)
- Questions appear one at a time — when you submit, next question loads immediately
- You do not have to wait for your opponent to finish a question to move to the next one

### 19.6 Scoring in a Battle

Correct answer: points based on difficulty (2/3/5 points as usual) PLUS a speed bonus
- If answered in first 10 seconds: +3 speed bonus points
- If answered in 10–20 seconds: +2 speed bonus points
- If answered in 20–30 seconds: +1 speed bonus point
- If answered after 30 seconds: no speed bonus

Wrong answer: 0 points (no negative marking in battles)

### 19.7 Battle Result Screen

After 5 minutes (or after both players answer all 10 questions, whichever is first):

Win Screen:
- "VICTORY!" text with trophy animation
- Points you scored vs points opponent scored
- Stars earned: your wager is returned + equal amount from opponent's wager
- XP earned: 50 XP + any bonus XP for accuracy
- "Rematch?" button and "Back to Lobby" button

Loss Screen:
- "DEFEAT" text (not harsh — encouraging message below)
- Encouraging quote: "Great effort! Every battle makes you stronger."
- Points you scored vs points opponent scored
- Stars deducted: only the wagered amount
- XP earned: 5 XP (participation reward — never punish for losing)
- "Rematch?" button and "Practice More" button

Draw Screen (if scores are equal):
- "DRAW" text
- Both players receive wagered Stars back (no gain, no loss)
- XP earned: 20 XP each

## SECTION 22 — ANTI-CHEAT RULES (NO COPY-PASTE, NO NEW TAB)

### 22.1 Where Anti-Cheat Applies

Anti-cheat rules apply in TWO places:
1. All Questions during normal learning (reduced rules)
2. Battle mode and Competition Events (full strict rules)

### 22.2 Anti-Cheat in Normal Learning

In regular day-by-day learning:
- Right-click context menu is disabled (cannot copy via right-click)
- Ctrl+C and Ctrl+V keyboard shortcuts are disabled on the question text area
- Text selection is disabled on question text (CSS: user-select: none)
- Tab switch detection: If user switches browser tabs, a yellow warning toast appears: "Hey! Complete this question without leaving the tab."
- After 3 tab switches on a single question: question is marked as skipped (no points awarded)
- None of this blocks tab switching for YouTube hint videos — the detection is paused when YouTube hint is clicked

### 22.3 Anti-Cheat in Battle Mode

During a live battle (full strict mode):
- Right-click completely disabled
- All keyboard shortcuts for copy/paste disabled
- Tab switching counts as answer submission (whatever is currently selected is submitted)
- Alt+Tab detection: switches to a "YOU LEFT THE BATTLE" screen, opponent gets a free correct answer point for each question skipped
- Browser minimize: treated same as tab switch
- Opening DevTools: Battle pauses, opponent is notified, match may be flagged for review
- All answers are server-validated — client-side cannot manipulate scores

### 22.4 Anti-Cheat in Competition Events (Safe Browser)

In the safe browser used for competition events:
- A separate browser-within-browser environment (sandboxed iframe)
- Opening a new tab: completely prevented (event handler blocks Ctrl+T)
- Copy-paste: completely blocked (all clipboard operations disabled)
- Right-click: completely disabled
- Browser back/forward: disabled within the competition view
- Refresh: triggers a confirmation — if confirmed, the user's answers reset
- DevTools: Opening DevTools causes immediate disqualification
- All keystrokes are logged for review by event organizers
- Session is tied to a unique token that cannot be shared

## SECTION 23 — SAFE BROWSER SYSTEM

### 23.1 What the Safe Browser Is

The Safe Browser is a special restricted viewing environment embedded within the EduQuest platform. It is activated ONLY when:
- A student clicks "Enter Competition" for an active competition event
- The student has already registered for that specific event

The Safe Browser is NOT the same as a regular browser window. It is a custom sandboxed environment within the EduQuest web app.

### 23.2 Technical Implementation of Safe Browser

The Safe Browser is implemented as a sandboxed iframe with strict security attributes:
- sandbox attribute: allows-scripts but disallows: allow-popups, allow-popups-to-escape-sandbox, allow-same-origin (for external requests), allow-top-navigation
- All keyboard shortcut events are intercepted at the document level and prevented
- Clipboard API is overridden to return null for all read operations
- Context menu (right-click menu) is prevented via oncontextmenu event listener
- Focus lock: The browser tab focus cannot leave the safe browser iframe without triggering detection
- Tab/window activity is tracked via Page Visibility API — hiding the tab is detected and logged

### 23.3 What Students Can Do Inside Safe Browser

- Read the competition question text
- Select answer options (MCQ)
- Type answers (Short answer / Long answer)
- Submit answers
- See their current score within the competition
- See the remaining time for the competition
- Scroll through long questions

### 23.4 What Students Cannot Do Inside Safe Browser

- Copy any text
- Paste any text
- Open a new tab
- Navigate to a different URL
- Use browser back/forward buttons
- Open browser developer tools
- Right-click anywhere
- Take screenshots using browser print function

- The system asks 5 questions to gauge level
- Based on performance, harder or easier questions are served next

- Provides a detailed "Your Proficiency Profile" report at the end

## SECTION 31 — STARS WALLET (VIRTUAL CURRENCY SYSTEM)

### 31.1 What Stars Are

Stars is EduQuest's virtual in-platform currency. It is NOT called "money" or "rupees" anywhere in the app — it is always referred to as "Stars" to maintain legal clarity. Stars are used exclusively within the EduQuest platform for battles and skill-based features.

Stars is legally classified as a virtual currency for platform use only — not as real money, not as a financial instrument. This avoids classification as real-money gambling.

### 31.2 How Stars Are Obtained

Students can obtain Stars in three ways:

Way 1 — Free Stars from Leveling Up:
- Every few levels award bonus Stars
- Level 3: +50 Stars
- Level 7: +100 Stars
- Level 9: +150 Stars
- Level 10 subscription: +Stars monthly gift
- These Stars are free and non-refundable

Way 2 — Purchasing Stars (Level 10+ only):
- Students can deposit real money into their Stars Wallet via Razorpay
- Stars are issued at a fixed rate: ₹1 = 10 Stars
- Minimum deposit: ₹50 (= 500 Stars)
- Maximum deposit: ₹500 at once
- The deposit interface is hidden behind the wallet section
- Clear legal notice displayed: "Stars are a virtual currency. Purchases are non-refundable. Stars have no real-world monetary value."

Way 3 — Winning Stars in Battles:
- When a player wins a wagered battle, they receive the opponent's wagered Stars
- If both players wagered 100 Stars, winner gets 200 Stars total (100 returned + 100 won)

### 31.3 How Stars Are Spent

Stars can only be used for:
1. Wagering in battle mode (Level 10+ only)
2. Future features (in-app cosmetics like profile borders, avatar accessories)

Stars CANNOT be used for:

## SECTION 40 — PERFORMANCE, CACHING & SCALABILITY

### 40.1 Frontend Performance

Next.js App Router Optimizations:
- Automatic code splitting — each page only loads its own JavaScript
- Image optimization via next/image — automatic WebP conversion, lazy loading, blur placeholder
- Font optimization via next/font — fonts loaded locally, no external requests
- Static generation for public pages (Home, Events listing, Leaderboard) — instant load
- ISR (Incremental Static Regeneration) for leaderboard — rebuilds every 24 hours

Performance Targets:
- LCP (Largest Contentful Paint): Under 2.5 seconds on 4G
- FID (First Input Delay): Under 100ms
- CLS (Cumulative Layout Shift): Under 0.1
- Time to Interactive: Under 3.5 seconds on 4G

### 40.2 Backend Performance

API Response Time Targets:
- Simple GET requests (fetch user data, fetch chapter list): Under 100ms
- Complex queries (leaderboard, battle history): Under 300ms
- Real-time battle events (Socket.IO): Under 50ms latency

Caching Strategy with Redis:
- User session data: Cached in Redis, 1 hour TTL
- Leaderboard rankings: Cached in Redis sorted sets, refreshed every 1 hour
- Course content (chapters, subtopics, questions): Cached for 24 hours
- Matchmaking queue: Stored in Redis, real-time

Database Optimizations:
- Indexes on: user_id, class_id, subject_id, chapter_id, level, streak
- Read replicas for analytics queries (so they don't slow down main database)
- Connection pooling via PgBouncer

### 40.3 Scalability Architecture

When user base grows:
- Horizontal scaling: Multiple NestJS instances behind a load balancer
- Redis Cluster for distributed caching
- Database read replicas for high-read operations
- CDN for all static assets (images, sounds, animations)
- Socket.IO with Redis adapter for multi-server real-time communication

## SECTION 41 — SECURITY ARCHITECTURE

### 41.1 Authentication Security

- Passwords: bcryptjs with salt rounds = 12 (never stored in plain text)
- Access tokens: JWT, 15-minute expiry
- Refresh tokens: Stored in httpOnly, Secure, SameSite=Strict cookies
- Token refresh: Silent rotation — new refresh token issued on every use
- OAuth: Google OAuth 2.0 with PKCE
- Rate limiting: Max 5 login attempts per IP per 15 minutes (then CAPTCHA)
- After 10 failed attempts: Account locked for 1 hour, email sent

### 41.2 API Security

- All APIs use HTTPS only (HTTP Strict Transport Security header enforced)
- CORS: Only the EduQuest frontend domain is whitelisted
- Helmet.js: Sets 15+ security headers automatically
- Input validation: Every API endpoint validates input using class-validator
- SQL injection: Prevented by Prisma ORM (parameterized queries)
- XSS: Next.js auto-escapes all rendered content; Content Security Policy header set
- CSRF: Not needed because JWTs stored in httpOnly cookies (not readable by JS)
- Rate limiting: Global (100 requests/minute per IP) and per-endpoint (tighter for sensitive routes)

### 41.3 Payment Security

- Razorpay webhook signature verified on server before processing any payment
- Payment amounts are always verified server-side — client sends an order ID, not amount
- Stars balances updated only after payment verification — never on client signal
- All payment data stored encrypted at rest

### 41.4 Data Privacy

- No user data sold to third parties
- Analytics data anonymized before processing
- Users can request account deletion (GDPR-style) — all data deleted within 30 days
- Passwords never emailed or logged
- Error logs sanitized to remove personal data before storage
